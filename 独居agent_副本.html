DATABASE DESIGN

数据库设计

数据结构如何支撑功能闭环

👤 profiles 表
user_id text 用户标识（MVP: "me"）
allergy_list text[] 过敏原数组
fear_list text[] 恐惧症数组
updated_at timestamp 最近更新时间
📔 diary 表
id bigserial 自增主键
user_id text 所属用户
date date 记录日期
content text 润色后的日记内容
emotion_tag text 情绪标签
设计原则：单用户 MVP 阶段使用 user_id = "me"，预留多用户扩展空间（RLS 策略）

WORKFLOW · 01

核心工作流详解（一）

记忆档案加载 & 意图识别

对话启动 → 读取档案

每次对话开始时，自动从 profiles 表 拉取用户的过敏史、恐惧症等专属信息，注入系统 Prompt。

意图识别 LLM

将用户输入分类为四种意图：

record_diary update_profile generate_summary chat

规则兜底代码节点

关键词强匹配防止 LLM 误判：检测到"遇到了"、"真开心"等词，强制归为 record_diary，确保关键记录场景不遗漏。

WORKFLOW · 02

核心工作流详解（二）

日记写入 & 档案更新

日记写入链路

润色 LLM（让文字更温暖）→ 写入代码节点（POST 到 Supabase diary 表）→ 温柔回复确认

档案更新链路

提取档案信息 LLM（强制输出 JSON）→ 更新代码节点（合并已有数据）→ 回复确认

技术
难点
LLM 输出格式不稳定，需要在代码节点中做 JSON 解析容错处理，防止格式错误导致写入失败

CHALLENGE · 01

关键困难与解决思路（一）

脏字符与认证问题

问题
HTTP 请求报错 `ascii codec can't encode`，Supabase 连接失败
根源
从网页复制的 Publishable key 携带 `\xa0`（不间断空格） 等隐形字符，肉眼无法识别
解决
使用"文本还原器"工具清理字符 + 记事本中转纯手工输入 + 重新生成 Supabase 密钥
排查启示：遇到编码报错，先检查输入源的字符纯净度，"看不见的字符"是常见陷阱
09 / 15

CHALLENGE · 02

关键困难与解决思路（二）

401 认证与 RLS 策略

问题
HTTP 请求持续返回 401 Unauthorized，数据无法写入
根源
混淆了 publishable key 和 service_role key 的使用场景；RLS（行级安全）策略未正确配置
解决
用 service_role key 临时绕过 RLS 验证，定位到策略配置问题 → 重建正确的 RLS 策略 → 换回安全的 publishable key
调试方法论：假设 → 验证（临时绕过）→ 定位根因 → 修复 → 恢复安全机制
10 / 15

CHALLENGE · 03

关键困难与解决思路（三）

意图识别不稳定

问题
用户说"今天遇到了喜欢的人"，LLM 误判为 `chat`，未触发记录流程
根源
LLM 对"分享心情"和"请求记录"的边界判断模糊，存在概率性误判
解决
增加"规则兜底"代码节点，基于关键词强匹配（"遇到了"、"好开心"等），100% 确保关键场景不遗漏，优先级高于 LLM 判断
工程哲学：AI 不是万能的，在关键路径上需要确定性的规则兜底，不能完全依赖概率模型

DIFFERENTIATION

产品独特性与竞品对比

数据主权、可定制性、仪式感 —— 三个核心差异

维度 独居小助手 ✨ 豆包 / 通用 AI
记忆方式 结构化档案（过敏/恐惧） 泛化偏好记忆，无结构化
交互模式 主动引导记录 被动响应
日记功能 结构化存储 + 月度盘点 无
数据归属 用户自有数据库（Supabase） 黑盒存储，数据不透明
知识库 可自定义 RAG 知识库 内置通用知识，不可定制
